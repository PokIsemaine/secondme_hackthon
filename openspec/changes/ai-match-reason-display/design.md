## Context

当前 AI 匹配理由返回单一字符串，`/api/circle/match` 的 prompt 要求 AI 返回 JSON 数组，每条记录包含 `reason` 字段为纯文本。前端 `CircleDiscovery.tsx` 将 `matchReason` 以 `text-xs italic text-purple-700` 样式渲染在卡片底部，视觉权重极低，用户几乎忽略。

匹配理由是"协作撮合交接点"——没有清晰、可信的理由，用户无法建立对 AI 判断的信任，也无法决定是否与对方建立连接。

## Goals / Non-Goals

**Goals:**
- 匹配理由分两级展示：第一级类型标签 + 第二级双向论点
- 默认折叠第二级，用户点击展开
- 高分匹配（85%+）详细展开，低分匹配简短展示
- AI 返回结构化 reason 对象而非单一字符串

**Non-Goals:**
- 不改变匹配分数计算逻辑
- 不修改帖子列表的展示形式
- 不涉及移动端适配（本次仅桌面端）
- 不实现用户反馈/纠错机制

## Decisions

### 1. AI 返回结构化 reason 对象

**决定**: AI prompt 改为要求返回结构化 JSON，包含 `complementType`、`thesis`（双向）、`detailLevel`。

**替代方案考虑**:
- 保留纯字符串，前端自己解析 → 解析逻辑脆弱，容易出错
- 前端做 NLG 二次加工 → 增加前端复杂度，语义可能失真
- **最终选择**: AI 直接输出结构化数据，保证语义完整性

**reason 对象结构**:
```json
{
  "complementType": "能力互补",
  "thesis": {
    "forYou": "你的技术架构 → 解决他选型困境",
    "forThem": "他的产品视角 → 弥补你需求表达短板"
  },
  "detailLevel": "detailed"
}
```

### 2. detailLevel 规则

**决定**: 由 AI 根据匹配分数动态决定 `detailLevel`：
- 分数 >= 85% → `detailLevel: "detailed"`
- 分数 < 85% → `detailLevel: "brief"`

**理由**: 高分匹配需要更有说服力的理由来促使用户采取行动；低分匹配简短带过即可，避免信息过载。

### 3. 前端两级折叠 UI

**决定**:
- 折叠状态：显示 `complementType` 标签 + `thesis.forYou`（单向，一行）
- 展开状态：显示完整双向论点 `thesis.forYou` + `thesis.forThem`

**UI 示意**:
```
折叠：
┌──────────────────────────────────────┐
│ 🔷 能力互补                     [85%] ▼│
│ "你的技术架构 → 解决他选型困境"        │
└──────────────────────────────────────┘

展开：
┌──────────────────────────────────────┐
│ 🔷 能力互补                     [85%] ▲│
│ ─────────────────────────────────────  │
│ "你的技术架构 → 解决他选型困境"        │
│ "他的产品视角 → 弥补你需求表达短板"   │
└──────────────────────────────────────┘
```

### 4. API 响应解析兼容

**决定**: `route.ts` 解析 AI 响应时，兼容两种格式：
- 新格式：结构化 reason 对象（优先）
- 旧格式：纯字符串 reason（fallback，保持向后兼容）

**理由**: 避免 AI prompt 调整后线上旧响应导致解析失败。

## Risks / Trade-offs

- **[Risk]** AI prompt 修改后，旧缓存响应可能格式不一致
  → **Mitigation**: 解析逻辑做 try-catch fallback，发现格式异常时回退到原字符串reason

- **[Risk]** 结构化 reason 对象字段缺失导致前端报错
  → **Mitigation**: 前端做防御性读取，字段不存在时降级显示

- **[Trade-off]** detailLevel 只分两档，85% 阈值是否最优
  → **Mitigation**: 作为可配置常量，后续可根据数据调整

## Open Questions

- 85% 阈值是否需要做成可配置？
- 类型标签（能力互补/认知互补/双向互助）是否需要更丰富的视觉样式？
- 展开状态是否需要过渡动画？

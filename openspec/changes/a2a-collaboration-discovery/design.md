## Context

本产品是 Next.js 全栈 Web 应用，以 Second Me OAuth 登录作为用户身份基础，以知乎两个指定圈子的真实讨论作为协作发现场景。核心架构采用"A2A 作为产品主舞台"模式：用户 Agent 与对方 Agent（或候选代理）先完成协商，再将结构化提案交给真人确认。

**当前状态**：项目已初始化 Next.js + Prisma + TypeScript 基础架构，已接入 Second Me OAuth 模块。

**约束条件**：
- 知乎仅开放两个白名单圈子 (2001009660925334090, 2015023739549529606)
- 知乎 Base URL 为 https://openapi.zhihu.com，采用 AK/SK 签名鉴权，全局限流 10 QPS
- 全网可信搜单用户总调用量上限 1000 次，必须做缓存
- Second Me 仅提供 Agent chat 接口，总结/判断/生成需应用侧 LLM

## Goals / Non-Goals

**Goals:**
- 实现 Second Me OAuth 登录并获取用户 Agent 调用能力
- 实现用户补位画像生成与管理
- 实现知乎圈子内容读取（仅限白名单圈子）
- 实现互补识别引擎（长板/短板标签识别）
- 实现候选代理生成器（基于公开内容构建对方画像）
- **实现双 Agent 协商引擎（核心）**，支持预协商与正式协商两阶段
- 实现协作提案生成器
- 实现"求补位发帖"功能
- 实现 Agent 协商室可视化页面

**Non-Goals:**
- 不实现全站知乎用户建模
- 不实现自动私聊或站外导流
- 不实现长期项目协作管理
- 不实现非白名单圈子扩展

## Decisions

### 决策 1: A2A 协商采用"双 LLM 调用 + 状态管理"方案

**选项 A**: 使用 MCP 协议实现真正的 Agent-to-Agent 通信
**选项 B**: 应用侧协调双 Agent 对话，通过消息队列管理协商状态

**选择 B**: 原因包括：(1) Second Me Agent Chat 接口是同步的，无法直接实现真正的 A2A 协议；(2) MVP 阶段需快速验证产品假设，应用层协调更灵活；(3) 可在协商室内可视化展示回合流，符合"让 A2A 成为主舞台"的产品理念。

### 决策 2: 候选代理采用"公开内容摘要 + LLM 推断"方案

**选项 A**: 使用预训练模型直接生成对方画像
**选项 B**: 基于对方在圈子的帖子/评论/互动数据，用 LLM 推断长板/需求

**选择 B**: 知乎公开数据有限，但足以推断基本能力标签。LLM 可从讨论内容中推断"可能擅长的领域"和"当前在意的问题"。

### 决策 3: 应用侧 LLM 选型

**选项 A**: 使用 OpenAI GPT-4
**选项 B**: 使用 Claude API
**选项 C**: 使用硅基流动等国产 API

**选择 C (待定)**: 需根据成本、响应速度、中国区支持等因素选择。可先用 Claude/GPT-4 原型验证，后续切换。

### 决策 4: 协商模板固定为 5 轮

**选项 A**: 开放式协商，让 Agent 自由发挥
**选项 B**: 固定 5 轮模板（定义问题 → 交换价值 → 确认边界 → 设计最小合作形式 → 建议继续/终止）

**选择 B**: PRD 明确协商需"遵循固定模板，避免空泛闲聊"。固定轮次确保输出可预期，也便于可视化展示。

### 决策 5: 知乎 API 缓存策略

**选项 A**: 实时调用，每次请求都访问知乎 API
**选项 B**: 关键数据做本地缓存，热榜做定时更新

**选择 B**: (1) 可信搜有 1000 次上限；(2) 圈子内容变化不频繁；(3) 10 QPS 限流需规避。实现 Redis 缓存层。

## Risks / Trade-offs

### 风险 1: A2A 协商感知度不足
[风险] 如果 Agent 协商只发生在后台，用户和评委感知不到 A2A 价值
[ mitigation ] 必须实现"Agent 协商室"页面，展示每轮发言、共识、分歧、可视化流程

### 风险 2: 对方真实 Agent 不可用
[风险] 不是所有人都会立刻授权登录，无法直接进入 Full A2A
[ mitigation ] 实现两阶段 A2A：先用候选代理做预协商，再在对方登录后升级为真实双 Agent 协商

### 风险 3: Second Me 能力边界误解
[风险] 评委可能误以为所有总结和判断都来自 Second Me
[ mitigation ] 在页面和文档中明确说明 Second Me 提供身份与 Agent chat，额外能力由应用侧 LLM 完成

### 风险 4: 知乎能力边界质疑
[风险] 容易被质疑是否在做全站抓取
[ mitigation ] 严格限制仅使用两个白名单圈子、热榜、可信搜，不做全站私域抓取

### 风险 5: 可信搜额度耗尽
[风险] 单用户 1000 次调用上限很快用完
[ mitigation ] 只在关键节点调用（提案生成前），实现缓存，并记录调用次数

### 风险 6: MVP 时间紧张
[风险] 8 个功能模块 + 6+ 个页面，开发周期紧张
[ mitigation ] 优先实现核心链路：登录 → 画像 → 圈子发现 → 互补识别 → 协商室 → 提案，发帖功能可作为可选

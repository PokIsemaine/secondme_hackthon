## Context

当前 Demo 页面 (`/demo`) 使用静态 mock 数据展示用户旅程，无法体现真实 API 调用体验。根据 PRD v2 第 11 节定义的 7 个核心页面，需要将 Demo 页面改造为使用真实 API 串联完整流程。

**当前状态**:
- Demo 页面使用 `src/lib/mock.ts` 中的静态数据
- 包含 5 个模拟帖子、2 个匹配结果、5 轮协商对话
- 不调用任何真实 API

**约束**:
- 用户需要先登录 Second Me 才能调用 API
- 知乎 API 有限流 (10 QPS) 和可信搜额度限制 (1000 次)
- 部分 API 需要有效的 access token

## Goals / Non-Goals

**Goals:**
- 修改 `/demo` 页面使用真实 API 调用
- 保持 Demo 的可演示性（未登录时展示引导）
- 对齐 PRD v2 定义的 6 步用户旅程

**Non-Goals:**
- 不修改现有 API 端点逻辑
- 不添加新的后端服务
- 不实现实时 WebSocket 通信

## Decisions

### 决策 1: Demo 页面登录态处理

**选项 A**: 强制要求登录后才能使用 Demo
**选项 B**: 未登录时展示引导，登录后调用真实 API
**选项 C**: 始终使用 mock 数据，仅展示流程

**选择 B**: 平衡了真实体验和演示便利性。未登录用户可以看到完整流程介绍，登录用户可以使用真实功能。

### 决策 2: API 调用策略

**选项 A**: 每次 Demo 访问都调用真实 API
**选项 B**: 使用 mock + 真实 API 混合模式
**选项 C**: Demo 模式下始终使用 mock，登录后切换真实

**选择 C**: 考虑到 API 限流和额度限制，Demo 模式使用 mock 数据保持演示稳定性，用户真实使用时调用真实 API。

### 决策 3: 页面架构

**选项 A**: 单个 `page.tsx` 文件包含所有逻辑
**选项 B**: 拆分为多个组件
**选项 C**: 使用子路由 (`/demo/intro`, `/demo/profile`, ...)

**选择 A**: Demo 页面逻辑相对简单，保持单一文件便于维护。使用 `useState` 控制步骤切换。

## Risks / Trade-offs

### 风险 1: API 调用失败导致 Demo 无法展示
[风险] 网络问题或 token 过期导致 API 调用失败
[mitigation] 添加错误边界和降级处理，失败时回退到 mock 数据展示

### 风险 2: 知乎 API 限流
[风险] 频繁访问触发 10 QPS 限流
[mitigation] Demo 页面添加请求间隔控制，避免连续快速调用

### 风险 3: 协商过程耗时过长
[风险] 5 轮 A2A 协商可能耗时 30+ 秒
[mitigation] 添加 loading 状态和超时处理，允许用户跳过实际协商查看结果

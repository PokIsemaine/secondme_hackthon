## Context

当前 Next.js 应用存在多个独立页面：
- `/` - 首页（登录、用户信息、圈子发现）
- `/demo` - 演示页面（完整流程）
- `/negotiation` - 协商室列表
- `/negotiation/[id]` - 协商室详情
- `/seeker` - 求补位发帖

PRD v2 要求单页面用户旅程，所有功能应在单一页面内完成。

## Goals / Non-Goals

**Goals:**
- 将 `/demo` 设为应用默认入口
- 将原首页的登录、用户画像、圈子发现功能融入 demo 页面
- 删除独立的协商室页面，保留协商功能在 demo 页面内
- 实现完整的单页面用户流程

**Non-Goals:**
- 不修改 API 路由结构
- 不改变数据库模型
- 不修改认证逻辑

## Decisions

### D1: 路由策略
**选择**: 使用 Next.js 重定向，将 `/` 重定向到 `/demo`

**理由**:
- 最简单直接，无需修改现有代码结构
- `/demo` 已实现完整功能链路
- 保留 `/demo` 路径可方便演示

**替代方案**:
- 将 demo 页面内容移到 `src/app/page.tsx` - 需要大量迁移，风险较高

### D2: 页面组件结构
**选择**: 在 demo 页面内添加登录状态检测和引导

**理由**:
- 现有 demo 页面已有 `isLoggedIn` 和 `demoMode` 状态
- 只需增加未登录状态下的引导 UI

**替代方案**:
- 创建新的 Layout 组件 - 改动过大

### D3: 协商功能位置
**选择**: 保留 `/api/negotiation/` API，删除 `/negotiation` 页面路由

**理由**:
- API 需要保留供 demo 页面调用
- 页面路由不再需要，用户在 demo 页面内完成协商

### D4: 求补位发帖入口
**选择**: 保持 `/seeker` 独立路由

**理由**:
- 可作为辅助入口，用户可在流程中跳转
- 不影响主流程的单页面体验

## Risks / Trade-offs

- **风险**: 用户可能习惯访问 `/` 但被重定向
  - **缓解**: 添加适当的引导文案说明

- **风险**: demo 页面代码可能变得复杂
  - **缓解**: 使用组件拆分，保持代码组织

## Migration Plan

1. 创建 `/` 的重定向到 `/demo`
2. 修改 demo 页面，添加登录引导 UI
3. 删除 `/negotiation` 和 `/negotiation/[id]` 路由目录
4. 验证完整流程可正常运行

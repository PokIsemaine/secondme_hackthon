## 一、产品概述

### 1.1 产品定位

**一句话描述**
「认知盲区拼图」是一款基于 Second Me 的 A2A 协作发现产品：它从知乎指定圈子的真实讨论中识别“谁能补上谁的短板”，并让双方 Agent 先完成一轮互补协商，再把结构化合作提案交给真人决定。&#x20;

### 1.2 核心价值主张

现有推荐系统更擅长告诉用户“你可能喜欢什么”，却不擅长告诉用户“你现在最需要谁”。本产品不做“相似匹配”，而做“互补匹配”；不止帮用户发现对象，还让双方 Agent 先替真人谈一轮，降低合作启动成本。&#x20;

### 1.3 产品形态

本产品是一个 **GUI 完整应用**，并以 **OAuth 登录后的用户 Agent** 作为核心参与者。Second Me 负责提供用户身份与分身对话能力；应用侧负责协商编排、内容理解、结构化总结与提案生成。&#x20;

---

## 二、项目背景

黑客松要求作品必须体现 A2A，而不是普通的 AI 助手。Second Me 当前明确支持 OAuth 登录、获取用户基本信息、调用用户 Agent；知乎开放了两个专属圈子的内容读写互动能力，以及热榜与全网可信搜，适合把 Agent 放进真实社区语境里。&#x20;

因此，本项目不把 Agent 仅仅当作“文案生成器”，而是把它设定为：

* 用户的前置表达代理
* 用户的需求/长板总结代理
* 双方协作可行性协商代理
* 协作提案生成代理

## 三、目标用户

### 3.1 核心用户

* 有明确专业长板的人
* 有现实问题待解决的人
* 不擅长主动社交，但愿意合作的人
* 愿意把一部分前置沟通交给 Agent 的用户&#x20;

### 3.2 典型用户

* 工程师：能提供技术判断，但缺产品抽象或表达支持
* 产品经理：能梳理需求，但缺技术架构判断
* 设计师：能提供视觉与体验，但缺增长或商业化视角
* 创作者：能提供内容，但缺产品化或技术支持&#x20;

## 四、问题陈述

用户在真实讨论中经常暴露出“问题”和“盲区”，也会展现“经验”和“长板”，但这些信号通常不会被转化为合作机会。用户即便发现潜在对象，也需要自己去试探、寒暄、解释价值，成本高且容易中断。传统社交产品更擅长找相似的人，不擅长找互补的人。

本产品试图解决的核心问题是：

**如何让“谁能补谁”的关系，在真实讨论里被识别出来，并由双方 Agent 先完成一轮低成本协商。**

## 五、产品目标

### 5.1 总目标

构建一个以 Agent 为核心参与者的 A2A 协作发现产品，让用户在知乎真实讨论中发现互补对象，并通过双方 Agent 的预协商，降低现实合作的启动门槛。

### 5.2 MVP 目标

验证三个核心假设：

1. 用户愿意让自己的 Agent 先替自己做前置表达与协商
2. 基于真实讨论语境的互补匹配，比基于兴趣相似的匹配更容易形成行动
3. 可视化的双 Agent 协商过程，比单纯“推荐一个人给你”更能促成真人确认

### 5.3 非目标

首版不验证：

* 全站级用户增长
* 长期学习效果提升
* 自动私聊或站外导流
* 复杂长期合作管理
* 全站知乎内容抓取&#x20;

## 七、核心理念

### 7.1 从“盲区识别”升级为“Agent 补位协商”

“认知盲区”只是入口，不是最终输出。
最终输出不应该是一张静态盲区图，而应该是一份 **由双方 Agent 协商后得出的合作提案**。

因此，本产品的真正核心不是：

* 你缺什么知识

而是：

* 你现在卡住的问题是什么
* 谁能补你的短板
* 你能交换什么价值
* 双方 Agent 谈下来，这次合作值不值得开始

### 7.2 A2A 的定义

本产品中的 A2A 不是“两个模型随便对话”，而是：

**两个代表不同用户立场、能力、边界和需求的 Agent，围绕一个具体合作问题进行协商。**

## 八、产品核心链路

### 8.1 主链路

**知乎圈子讨论 → 识别互补对象 → 拉起我的 Agent 与对方 Agent/候选代理 → 进行多轮协商 → 输出合作提案 → 真人确认是否继续**

这条链路建立在你原来“圈子内容 → 识别长板/短板 → 发现互补对象 → Agent 双向分析 → 输出合作提案”的基础上，但把 **A2A 协商过程** 提升为核心舞台。&#x20;

### 8.2 两阶段 A2A 结构

#### 阶段一：预协商（Semi-A2A）

当对方还没有登录应用、无法调用真实 Second Me Agent 时：

* 我的 Agent：真实 Second Me Agent
* 对方 Agent：由知乎公开内容构建的“候选代理”

系统先进行一轮“预协商”，判断：

* 是否互补
* 是否值得进一步连接
* 推荐的合作形式是什么

#### 阶段二：正式协商（Full A2A）

当对方接受邀请并登录后：

* 我的 Agent：真实 Agent
* 对方 Agent：真实 Agent

系统发起正式双 Agent 协商，输出更准确的合作提案。

## 十、MVP 功能设计

### 功能 1：Second Me OAuth 登录

用户点击“使用 Second Me 登录”，完成 OAuth 后进入应用。系统获取：

* 用户 Second Me 身份
* 基本信息
* 用户 Agent 调用能力 &#x20;

### 功能 2：我的补位画像

输入方式：

* 用户主动填写
* 用户与自己 Agent 的盘点式对话
* 圈子互动上下文作为辅助信号&#x20;

输出字段：

* 我擅长什么
* 我最近卡在哪
* 我愿意提供什么
* 我希望得到什么
* 我的合作边界
* 画像可信度

### 功能 3：圈子内容发现

系统读取两个指定圈子的帖子、评论与互动信号，展示帖子流、评论摘要、热门讨论标签和“发现互补机会”按钮。&#x20;

### 功能 4：互补识别引擎

系统基于讨论中的公开信号识别两类标签：

* **能力长板**：经验、判断、结构化表达、持续高质量输出
* **认知短板**：求助、追问、困惑、明显缺口、资源交换意图&#x20;

输出：

* 长板标签
* 短板标签
* 协作场景标签
* 置信度

### 功能 5：候选代理生成器

当对方尚未登录时，系统根据其知乎公开内容构建“候选代理画像”，包括：

* 可能擅长的领域
* 可能当前在意的问题
* 沟通风格线索
* 合作意愿推测
* 画像可信度

该候选代理只用于预协商，不视为真实授权 Agent。

### 功能 6：双 Agent 协商引擎

这是本产品的核心功能。

#### 输入

* 用户 A 的真实 Second Me Agent
* 用户 B 的真实 Agent，或候选代理
* 当前讨论上下文
* 协商目标模板

#### 协商议题

1. 当前要解决的具体问题是什么
2. A 能提供什么
3. B 能提供什么
4. 这次合作最小可执行形式是什么
5. 双方边界是什么
6. 是否值得真人继续投入

#### 输出

* 协商回合记录
* 共识摘要
* 分歧点摘要
* 推荐合作形式
* 建议投入时长
* 是否建议真人接入

### 功能 7：协作提案生成

在协商结束后，系统生成一份结构化提案，内容包括：

* 协作主题
* 互补原因
* 双方各自提供的价值
* 推荐开始方式
* 风险与边界
* 建议下一步行动

### 功能 8：求补位发帖

用户填写：

* 当前卡住的问题
* 能提供的交换价值
* 希望获得的帮助

系统先由 Agent 生成发帖草稿，再发布到指定圈子。发帖仅支持两个白名单圈子。&#x20;

## 十二、用户旅程

### 旅程 A：从真实讨论到 Agent 协商

1. 用户通过 Second Me 登录
2. 完成补位画像
3. 浏览知乎指定圈子讨论
4. 点击“发现互补机会”
5. 系统识别潜在互补对象
6. 生成对方候选代理
7. 发起预协商
8. 展示 Agent 协商过程
9. 输出合作提案
10. 真人决定是否继续

### 旅程 B：从发帖求助到双 Agent 协商

1. 用户通过 Second Me 登录
2. 填写“我卡住的问题”和“我能交换的价值”
3. Agent 生成发帖草稿
4. 发布到指定圈子
5. 评论中出现潜在补位者
6. 系统识别互补关系
7. 拉起我的 Agent 与对方候选代理
8. 输出协商结果
9. 若对方接受并授权，则进入真实双 Agent 协商

## 十三、识别与协商逻辑

### 13.1 长板识别逻辑

系统将以下信号视为长板线索：

* 持续提供经验、方案、判断
* 使用结构化表达解释问题
* 多次高质量回应追问
* 展现专业术语密度与上下文理解
* 获得正向互动反馈&#x20;

### 13.2 短板识别逻辑

系统将以下信号视为短板线索：

* 明确求助
* 表达不会做、看不懂、卡住
* 连续追问某类问题
* 对某类内容高度关注但缺解决方案
* 明确表达资源或能力不足&#x20;

### 13.3 互补匹配逻辑

系统不算“相似度”，而计算：

* 我的长板是否补你的短板
* 你的长板是否补我的短板
* 双方是否处于同一问题语境
* 是否存在现实可执行的协作形式
* 是否值得用一次低成本合作验证

### 13.4 A2A 协商逻辑

协商遵循固定模板，避免空泛闲聊：

* Round 1：定义问题
* Round 2：交换可提供价值
* Round 3：确认边界
* Round 4：设计最低成本合作形式
* Round 5：给出继续或终止建议

## 十五、热榜与可信搜的角色

### 15.1 热榜角色

热榜不是主匹配数据源，而是帮助 Agent 感知当前中文互联网话题背景，让提案和发帖更贴近现实讨论语境。

### 15.2 可信搜角色

可信搜用于关键节点增强：

* 在提案生成前补充相关讨论依据
* 让 Agent 回应更有内容支撑
* 用 `authority_level` 筛选高质量信息

由于总调用量有限，必须缓存，不可作为高频实时依赖。

## 十八、风险与对策

### 风险 1：A2A 不够显性

**问题**：如果 Agent 协商只发生在后台，评委感知不到。
**对策**：必须做“Agent 协商室”页面，展示回合流、共识、分歧和真人确认按钮。

### 风险 2：对方真实 Agent 不可用

**问题**：不是所有人都会立刻授权登录。
**对策**：采用“两阶段 A2A”，先用候选代理做预协商，再在对方登录后升级为真实双 Agent 协商。

### 风险 3：Second Me 能力被误解为万能

**问题**：评委可能误以为所有总结和判断都来自 Second Me。
**对策**：明确说明 Second Me 提供用户身份与 Agent chat，额外总结/结构化能力由应用侧 LLM 完成。

### 风险 4：知乎能力边界不清

**问题**：容易被质疑是不是在做全站抓取。
**对策**：明确只使用两个指定圈子、热榜和可信搜，不做全站私域抓取。

### 风险 5：可信搜额度不够

**问题**：单用户总调用量 1000 次，用完即止。
**对策**：只在关键节点调用，并做缓存。
